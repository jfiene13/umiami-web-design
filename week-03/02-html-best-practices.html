HTML Best Practices

If you read Zeldman (You did read Zeldman, I hope...) most of this should be familiar.

Use the proper DOCTYPE

<!DOCTYPE html>

There are several DOCTYPES, but we'll be using the one that tells the browser our code is written in HTML5.

Declare your character set

<meta charset="UTF-8">

UTF-8 is recognized as a good standard because it contains over 1 million individual characters.

Close your tags

Good: <p>Text</p>

Bad: <p>Text

There are a few exceptions to this. <br />, <hr /> and <img /> are among a short list of self-closing tags. Can you name others?

This is a good habit to develop before you begin working with CSS and JavaScript.

Write your tags (elements, attributes) in lowercase

<p class="intro">Text</p>

HTML is not case sensitive, but this is an accepted best practice.

Use quotes around attribute values

<img alt="Sebastian the Ibis" class="photo" src="photo.jpg" />

Another accepted best practice.

All attributes should have values

<hr noshade="noshade" />

Most attributes require a value. Some don't. Again, it's a good habit to develop before you begin working with CSS and JavaScript.

Do not use double dashes inside a comment

<!-- This is what a comment looks like. -->

<!-- If you must use double dashes, separate them by a space - - like so. -->

Double dashes belong at the beginning and end of a comment. Putting them within a comment will invalidate code.

Encode special characters

If you read the source code of this document, you will see funky code that starts with an ampersand (&) and ends with a semicolon (;). These are special characters.

Special characters are predominantly used in foreign languages. There are also symbols we can use such as:

  • a double dagger: ‡
  • a spade: ♠
  • a heart: ♥
  • a sterling pound: £
  • a euro: €
  • copyright: ©

WebMonkey has a cheat sheet, although this is by no means complete.

Indent your code, two spaces for each level

HTML is not sensitive to white space. Following this tip will help keep code managable and readable. Spacing between lines of code is a matter of personal preference. Find something that you are comfortable with and remain consistent.
